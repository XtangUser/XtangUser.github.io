Dockerfile 注释：

  • FROM：指定基础镜像，此处选用轻量级的 node:14-alpine。
  • WORKDIR：设置容器内的工作目录，后续命令在该目录下执行。
  • COPY：将本地文件复制到容器内。先复制 package.json 可充分利用 Docker 缓存，避免代码变动时重复安装依赖。
  • RUN：在构建过程中执行命令，此处运行 npm install 安装依赖。
  • EXPOSE：声明容器运行时监听的端口（仅文档说明，实际映射需在运行或 compose 中指定）。
  • CMD：容器启动时执行的命令，此处为 npm start。

配置说明：

  • location /：处理根路径请求，将静态文件返回。try_files 用于 SPA 路由回退，这里简单使用。
  • location /api/：匹配 /api/ 开头的请求，通过 proxy_pass 转发到后端服务 backend:3000（注意 backend 是 Docker Compose 中定义的服务名，Docker 内部 DNS 会自动解析）。
  • proxy_set_header 设置转发请求头，使后端能获取真实客户端信息。

docker-compose.yml 注释：

  • version：Compose 文件格式版本，使用 3.8 支持较新特性。
  • services：定义两个服务：
    • backend：从 ./backend 目录构建镜像，容器名为 backend，restart: always 保证服务异常退出后自动重启。加入自定义网络 app-network。
    • frontend：从 ./frontend 目录构建，映射宿主机 80 端口到容器 80 端口，通过 depends_on 确保后端先启动（但不会等待后端完全就绪，仅控制启动顺序）。同样加入 app-network。
  • networks：创建一个桥接网络 app-network，使两个容器可以相互通信（通过服务名解析）。

二、核心组件详解

2.1 Docker 网络层

网络类型：Bridge（桥接网络）

原理：

  • Docker 在宿主机创建一个虚拟网桥 docker0
  • 每个容器分配独立的虚拟网卡（veth pair）
  • 容器间通过网桥通信，实现网络隔离
  • 容器通过服务名（如 backend）互相解析（内置 DNS）

2.2 前端容器架构

原理说明：

  1. 多阶段构建：直接使用官方 Nginx 镜像，无需额外配置
  2. 配置覆盖：default.conf 覆盖默认配置，实现自定义路由
  3. 静态文件服务：Nginx 作为静态文件服务器，性能优异
  4. Alpine 基础：体积小（~20MB），启动快，安全性高

2.3 后端容器架构

原理说明：

  1. 层缓存优化：package.json 先复制，依赖层可复用
  2. 工作目录：/app 作为应用根目录
  3. 进程管理：Node.js 作为主进程（PID 1）
  4. 信号处理：直接接收 Docker 的 SIGTERM 信号

2.4 Nginx 反向代理配置

nginx.conf 详解

配置原理：

指令 作用 原理
proxy_pass 转发请求到后端 Nginx 作为反向代理，将 /api/ 请求转发到 http://backend:3000/api/
proxy_set_header 设置转发头 保留原始客户端信息，便于后端获取真实 IP
try_files 文件查找 先找静态文件，找不到则返回 index.html（SPA 支持）
location /api/ 路径匹配 前缀匹配，所有 /api/ 开头的请求走代理

架构优势

核心原理总结

组件 角色 关键技术
Docker 容器化引擎 Namespace, Cgroups, UnionFS
Docker Compose 编排工具 YAML 配置，服务编排
Nginx 反向代理 负载均衡，静态文件服务
Bridge 网络 容器通信 veth pair, iptables NAT
端口映射 外部访问 DNAT 规则
